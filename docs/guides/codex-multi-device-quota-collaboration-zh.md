# Codex 多设备额度协作

## 这项功能解决什么问题

同一个 Codex 账号同时登录在多台电脑时，官方速率窗口是账号共享的。单台设备看到的本地 token 日志只能说明该设备经 CCSwitchMulti 网关发出的请求，不能代表账号总消耗，也不能被换算成官方窗口额度。

多设备额度协作把这两类数据分开处理：

- 官方窗口：读取每台已接入设备最近一次查询到的官方 utilization 百分比，并按同一窗口取最高值，避免因为某台设备数据旧而低估消耗。
- 设备用量：汇总每台已接入设备经 CCSwitchMulti 网关记录的今日和最近 7 天 token、请求数，用于判断消耗主要来自哪里。
- 约束：可在官方窗口接近耗尽时，拒绝本机 CCSwitchMulti 网关继续转发 Codex 请求。

它不是账号级的全局流量控制器。没有经过 CCSwitchMulti 网关的原生 Codex App、CLI 或其它客户端请求不能被本机拦截，仍会消耗同一账号的官方窗口。

## 接入前准备

在每一台要参与协作的设备上完成以下条件：

1. 安装支持多设备额度协作的 CCSwitchMulti 版本。
2. 用同一个 Codex 账号登录。功能只会合并同一账号的报告。
3. 在 CCSwitchMulti 的设置中启用 WebDAV 或 S3 同步。所有参与设备必须使用同一个后端和同一个远端根目录。
4. 让要纳入统计和约束的 Codex 请求经过该设备的 CCSwitchMulti 本地网关。

不要把不同 Codex 账号共用同一个同步目录当成协作方式。程序会用账号标识生成隔离命名空间，因此不同账号的设备报告不会合并；但为降低运维混淆，仍建议为不同团队或账号使用不同的同步 profile 或根目录。

## 首次配置

以下操作需要在每一台设备各做一次。

1. 打开 CCSwitchMulti 的 Codex 用量与重置额度页面，点击刷新，确认能看到当前账号的官方窗口。
2. 在多设备额度协作卡片中设置便于识别的设备名称，例如 `办公室 Windows`、`家里 Mac` 或 `CI Linux`。
3. 先保持观测模式，点击保存协作设置。
4. 点击同步设备报告。首次同步会先上传本机的脱敏报告，再读取同账号下其它设备的报告。
5. 到下一台设备重复步骤 1 到 4。返回任意已配置设备后再次点击同步设备报告，应在设备列表中看到多台设备。

设备名称只用于展示；稳定的设备 ID 由本机设置自动生成。不要通过复制设置文件来“克隆”一台设备，否则两台机器可能共享同一个设备 ID 并互相覆盖报告。

## 如何验证协作正常

完成两台设备接入后，任意一台设备执行一次同步，并检查：

- 设备列表中至少有两条记录，且每条的最后上报时间会随着各设备同步而更新。
- 今日 token、7 天 token 和请求数只反映已接入网关的本地日志。没有流经网关的流量不会出现在这些数字中。
- 官方窗口仍作为单独的百分比卡片显示。多设备协作只取各设备报告中更高的官方利用率，不会把 token 数字换算成官方百分比。
- 将一台设备断网或停止同步超过 10 分钟后，该设备的旧官方窗口快照不会再触发本机约束，避免因为陈旧数据误拦截。

如果列表仍只显示本机，先确认另一台机器已成功刷新官方额度并点击同步设备报告，再检查两台机器的 WebDAV/S3 后端和远端根目录是否完全一致。

## 观测模式与约束模式

### 观测模式

观测模式只展示跨设备报告，不改变任何请求。建议先运行一段时间，确认设备列表、同步时间和官方窗口读数符合预期。

### 约束模式

约束模式会检查最近 10 分钟内的官方窗口快照。对于每个官方窗口，程序使用所有已同步设备中最高的 utilization；当窗口剩余百分比不高于你设置的阈值时，本机 CCSwitchMulti 网关会向后续 Codex 请求返回 HTTP 429。

例如阈值为 20%，某个官方窗口的最高利用率达到 82%，剩余约 18%，则本机经过 CCSwitchMulti 的 Codex 请求会被拒绝。

启用前请确认：

- 每台需要限制的设备都通过自己的 CCSwitchMulti 网关访问 Codex。
- 每台设备都定期刷新官方额度并同步报告；只同步一次无法提供持续保护。
- 未经过 CCSwitchMulti 的原生 Codex App、CLI、第三方客户端或绕过本地代理的请求不受限制。

约束模式不会自动降级模型、重写请求或替你关闭其它设备。它的行为是明确且可预期的本地网关拒绝。

## 隐私与远端数据

每台设备上传一个独立 JSON 报告，路径形式为：

```text
{remote_root}/quota-collaboration/v1/{account_scope}/{device_id}.json
```

报告只包含：

- 不可逆散列后的账号范围标识；
- 随机稳定设备 ID 和用户设置的设备名称；
- 上报时间；
- 今日和最近 7 天的聚合 token、请求数；
- 官方窗口名称、利用率百分比和重置时间。

报告不会包含 OAuth access token、refresh token、真实账号 ID、提示词、原始会话 JSONL、请求正文、工作目录或本机文件路径。

同步后端仍由你自己的 WebDAV 或 S3 凭据保护。建议为协作创建权限最小化的专用目录或 bucket 前缀，并只向参与设备授予读写权限。

## 常见问题与恢复

### 提示“请先刷新一次 Codex 官方额度”

先在该设备的 Codex 用量页面点击刷新并确认查询成功，再同步。协作报告需要官方窗口快照作为判断依据。

### 提示未配置同步

在设置中启用 WebDAV 或 S3，并确保远端根目录和其它设备一致。配置同步本身与额度协作共用后端，但额度报告不会覆盖数据库配置快照。

### 设备列表里有旧设备

旧设备最后一次报告会保留在后端和本地缓存中用于展示，但超过 10 分钟的官方窗口数据不会参与约束。若要彻底移除，删除同步后端中该设备 ID 对应的 JSON 文件，然后在任意设备重新同步。

### 不想继续限制请求

把协作模式切回观测模式并保存。网关会立刻停止基于额度协作返回 429；已记录的脱敏历史报告不会影响正常转发。

### 两台设备互相覆盖或始终显示为同一台

检查是否复制了 CCSwitchMulti 的设置文件或配置目录。每台设备必须有不同的 `deviceId`。为受影响设备重新生成独立设置后，刷新官方额度并重新同步即可。

## 运行建议

- 日常只需在开始高强度工作前刷新并同步一次；长时间多机并行时，建议每台设备周期性刷新官方额度并同步。
- 先用观测模式观察一到两个官方窗口，再按团队可接受的余量设置阈值。常见的保守起点是 20%。
- 当本机收到约束 429 时，不要把它误判为上游故障。先检查协作页面的最高官方窗口利用率、最近同步时间，以及是否有其它设备仍在绕过 CCSwitchMulti 消耗额度。
