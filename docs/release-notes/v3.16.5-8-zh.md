# CCSwitchMulti v3.16.5-8

本版新增 Codex 多设备额度协作，并修复商汤 SenseNova 协议误判、停用路由旧模型回退到官方额度等问题。

## 多设备额度协作

- Codex 使用页新增多设备额度协作面板，可通过 WebDAV 或 S3 交换同一账号下各设备的脱敏额度报告。
- 报告按设备独立存储，只包含不可逆账号 scope、设备标识、官方窗口利用率和本地 token/请求聚合，不上传 OAuth 凭据、prompt、工作目录或原始会话。
- 支持只读观测和可选约束模式。约束模式根据最近 10 分钟内各设备报告的最高官方窗口利用率判断；达到阈值时，仅拒绝经过本机 CCSwitchMulti 代理的 Codex 请求。
- 新增设备接入引导、约束确认和中文配置教程，明确原生直连 Codex 流量不受本机代理策略控制。

## Codex OAuth 来源身份

- 可信本地 Codex 请求携带唯一的官方 first-party `originator` 时保留原值，覆盖 CLI、TUI、VS Code、Desktop/ChatGPT Desktop 等已知客户端。
- 缺失、未知或重复来源值继续回退为 `codex_cli_rs`。
- External API 和协议转换入口不能继承调用方伪造的来源身份，继续使用受控回退值。

## 商汤 SenseNova

- 修复 MultiRouter 物化目标 Provider 时丢失 route `apiFormat` 的问题；route 的显式协议现在会覆盖目标 Provider 的陈旧元数据。
- 将 `sensenova.cn` 加入旧配置的 Chat Completions-only 识别，避免把 Codex Responses 请求原样发送到商汤并触发 `HTTP 400 invalid_tool_call_id`。
- 继续复用通用 Responses -> Chat 工具调用转换，不引入商汤专用 tool call ID 改写。

## MultiRouter 路由回退

- Codex Desktop 若暂存了已停用 route 的旧模型 alias，请求现在会明确 route miss，不再静默落到 `defaultRouteId=official`。
- 避免用户在同一会话切换到旧中转 alias 后，错误地继续看到官方额度耗尽提示。
- 新增连续模型切换回归测试，确认每次请求仍按当前 `body.model` 重新解析 route。

## 验证

- Codex provider、handler、Responses -> Chat 和流式转换测试通过。
- 多设备报告校验、数据库隔离、WebDAV/S3 同步和约束模式测试通过。
- TypeScript、Prettier、rustfmt、Clippy 与 GitHub CI 等价检查纳入发布门槛。
